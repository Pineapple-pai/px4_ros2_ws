# px4_ros2_ws 改造执行指南

本文档按你当前工作区的真实结构重写，目标是让 `px4_ros2_ws` 在保留原有 `ROS2 Autonomy` 模式的同时，新增一条 `FAST-LIO2 + Ego-Planner + PX4 Offboard` 链路，实现：

1. QGC 点击 `Go to location`
2. 目标被桥接到 Ego-Planner
3. Ego-Planner 基于点云和里程计实时重规划
4. `px4_trajectory_interface` 将规划结果转换成 PX4 Offboard 轨迹指令
5. 无人机在 Offboard 模式下绕障飞向目标点

---

## 一、先说结论：原始指南里需要修正的点

### 1.1 技术路线上的关键修正

原始指南有 5 个关键偏差，若不修正，落地时会直接卡住：

1. 当前仓库不是 `MAVROS` 方案，而是 `px4_msgs + Micro XRCE-DDS + px4_ros2_interface_lib`。
2. 当前仓库里已经有 `px4_autonomy_bringup`、`px4_fastlio_bridge`、`px4_nav2_bridge`，不需要重新设计这些包名。
3. Ego-Planner 在这个工作区里的实际输出不是 `MultiDOFJointTrajectory`，而是：
   - `traj_utils/msg/Bspline`
   - `quadrotor_msgs/msg/PositionCommand`
4. QGC 的地图点击不会天然变成 Ego-Planner 目标点，必须加一个 `VehicleCommand -> PoseStamped` 的目标桥。
5. 现有 FAST-LIO 桥里最容易出错的是坐标系，不是简单 `FLU -> FRD`，而是：
   - 世界坐标：`ENU -> NED`
   - 机体系：`FLU -> FRD`

### 1.2 Mid360 适配上的重点

Mid360 这条线有 4 个必须单独确认的地方：

1. 主机网卡必须和雷达处于同一网段。
2. `livox_ros_driver2` 的 `user_config_path` 必须指向正确的 Mid360 JSON。
3. FAST-LIO2 的 `lid_topic` / `imu_topic` / `scan_line` / `blind` 需要匹配 Mid360。
4. QGC 目标点、Ego-Planner 局部地图、PX4 本地坐标原点必须在起飞前对齐。

## 三、总体架构

### 3.1 保留双模式

改造后保留两条控制链：

1. 备用链路：`ROS2 Autonomy`
   用于你现有的任务模式、航点、距离避障逻辑。
2. 新链路：`Offboard + Ego-Planner`
   用于 QGC 点击地图后的实时重规划绕飞。

两者并存，但不要同时接管飞行。

### 3.2 新链路数据流

```text
QGC Go to location
  -> /fmu/out/vehicle_command (DO_REPOSITION)
  -> qgc_reposition_goal_bridge
  -> /move_base_simple/goal
  -> ego_planner_node
  -> /planning/bspline
  -> traj_server
  -> /planning/position_cmd
  -> px4_trajectory_interface
  -> /fmu/in/offboard_control_mode
  -> /fmu/in/trajectory_setpoint
  -> PX4 Offboard
```

同时，感知链路为：

```text
Livox Mid360
  -> livox_ros_driver2
  -> /livox/lidar + /livox/imu
  -> FAST-LIO2
  -> /Odometry + /Laser_map + /cloud_registered
  -> lio_odometry_bridge
  -> /autonomy/lio_odometry + /autonomy/local_map
  -> Ego-Planner
```

这条链虽然仍然不是“真正的 Mid360 雷达仿真”，但已经明显比 `2D RPLidar -> 仿真桥` 更接近真机的 3D 建图输入条件。

## 3. 轨迹转换节点与 Offboard

### 3.1 关键代码

新包：

- [src/px4_trajectory_interface](/home/p/px4_ros2_ws/src/px4_trajectory_interface)

- [trajectory_interface.cpp](/home/p/px4_ros2_ws/src/px4_trajectory_interface/src/trajectory_interface.cpp)

它做的事情：

1. 订阅 `/planning/position_cmd`
2. 将 Ego-Planner 输出从 `ENU` 转到 PX4 `NED`
3. 连续发布：
   - `/fmu/in/offboard_control_mode`
   - `/fmu/in/trajectory_setpoint`
4. 在 warmup 计数满足后自动请求 Offboard

### 3.2 ENU ↔ NED 变换规则

位置、速度、加速度：

```text
NED.x = ENU.y
NED.y = ENU.x
NED.z = -ENU.z
```

偏航：

```text
yaw_ned = pi/2 - yaw_enu
```

偏航角速度：

```text
yaw_rate_ned = -yaw_rate_enu
```

### 3.3 现有 FAST-LIO 桥的修正

已修正：

- [lio_odometry_bridge.py](/home/p/px4_ros2_ws/src/px4_fastlio_bridge/px4_fastlio_bridge/lio_odometry_bridge.py)

修正内容：

1. 世界坐标从 `ENU -> NED`
2. 姿态从 `ENU/FLU -> NED/FRD`
3. 机体 twist 继续保持 `FLU -> FRD`

这次不删原有模式，原因有两个：

1. 你明确要求保留备用链路

建议做法：

1. `autonomy_stack.launch.py` 继续保留

这比“一上来就删状态机”风险更低。

#### 阶段 C：规划链

```bash

```

通过标准：

- 有稳定输出，通常 > 10 Hz

在当前工作区的 Gazebo 仿真联调中，已经验证过以下最小闭环：

```bash
ros2 topic pub --once /move_base_simple/goal geometry_msgs/msg/PoseStamped \
  "{header: {frame_id: 'world'}, pose: {position: {x: 2.0, y: 0.0, z: 1.5}, orientation: {w: 1.0}}}"
```

预期现象：

1. `ego_planner_node` 打印 `Triggered!`
2. 状态机出现 `WAIT_TARGET -> GEN_NEW_TRAJ -> EXEC_TRAJ`
3. `/planning/position_cmd` 开始输出
4. `/fmu/in/trajectory_setpoint` 开始输出

其中已验证的一组示例输出特征是：

- `/planning/position_cmd.position ~= (2.01, 0.00, 1.51)`
- `/fmu/in/trajectory_setpoint.position ~= (0.00, 2.01, -1.51)`

这正对应了 `ENU -> NED` 变换：

```text
(x_enu, y_enu, z_enu) -> (y_ned, x_ned, -z_ned)
```

#### 阶段 D：Offboard 接管

检查：

```bash
ros2 topic echo /fmu/out/vehicle_status_v4 --once
```

通过标准：

- `nav_state = 14` 表示已进入 Offboard

#### 阶段 E：绕障效果

通过标准：

1. 无障碍时基本直飞
2. 障碍位于直线路径时出现明显绕行
3. 障碍靠近时仍保持连续 setpoint，不出现长时间停摆

## 六、最容易出错的地方

### 6.1 QGC 点击了，但规划器没有反应

先查：

```bash
ros2 topic echo /fmu/out/vehicle_command --once
ros2 topic echo /move_base_simple/goal --once
```

原因通常是：

1. 没收到 `VEHICLE_CMD_DO_REPOSITION`
2. `require_external_command` 过滤掉了消息
3. PX4 还没有有效 `ref_lat/ref_lon`

### 6.2 有目标点，但没有 `/planning/position_cmd`

通常是：

1. `/autonomy/lio_odometry` 没数据
2. `/autonomy/local_map` 没数据
3. Ego-Planner 局部地图尺寸过小
4. `flight_type` 不适合当前模式

### 6.3 进入 Offboard 后飞向错误方向

优先怀疑：

1. `ENU -> NED` 没做好
2. LIO map 原点和 PX4 local 原点没对齐
3. 偏航用了 ENU 公式直接喂给 PX4

### 6.4 Mid360 有点云但 FAST-LIO 发散

优先排查：

1. `imu_topic` 是否真的是 `/livox/imu`
2. `blind` 是否过大
3. 外参是否错写
4. 雷达安装方向与 `extrinsic_R` 是否一致

### 6.5 仿真能飞，真机不能飞

高概率原因：

1. PX4 本地原点和 LIO 原点不一致
2. QGC 目标转换使用了 PX4 global ref，但 LIO world 没对齐起飞点
3. 网卡 IP 和 Mid360 不在同网段
4. `offboard_control_mode` 发送频率过低

---

## 七、建议的执行顺序

强烈建议按下面顺序推进，不要一步到位：

1. 先编译新包，确保无编译错误
2. 只跑 `FAST-LIO2 + lio_odometry_bridge`
3. 再跑 `Ego-Planner`，只看 `/planning/position_cmd`
4. 再接 `px4_trajectory_interface`
5. 最后再用 QGC 地图点击测试整链路

## 八、这次改造后你该怎么用

### 备用模式

原模式仍然保留：

```bash
ros2 launch px4_autonomy_bringup autonomy_stack.launch.py
```

### 新规划模式

```bash
./scripts/start_autonomy_with_planning.sh
```

如果这条链路调通，你的项目就已经具备了和 `LIO-Drone-250` 同类的核心能力：

1. QGC 下发目标
2. Ego-Planner 重规划
3. Offboard 连续轨迹跟踪
4. 基于点云的实时绕障

但要特别注意一件事：

真正决定“看起来像不像 LIO-Drone-250”的，不是 launch 是否能启动，而是 `LIO map 原点、QGC 局部目标、PX4 local frame` 这三者是否对齐。这个问题如果没处理好，系统会“能动，但飞错方向”。
